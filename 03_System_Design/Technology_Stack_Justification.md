# เหตุผลในการเลือกใช้เทคโนโลยี (Technology Stack Justification)

เอกสารนี้อธิบายเหตุผลเบื้องหลังการเลือกเทคโนโลยีสำหรับโครงการ "Wording"

## 1. ภาพรวมสถาปัตยกรรม
สถาปัตยกรรมที่เลือกใช้คือ **Serverless Web Application** โดยมีส่วนประกอบหลักดังนี้:
- **Frontend:** Single Page Application (SPA)
- **Backend:** Firebase (Backend as a Service - BaaS)

## 2. เทคโนโลยีที่เลือกใช้

### Frontend
| เทคโนโลยี | เหตุผลในการเลือก |
| :--- | :--- |
| **HTML5** | เป็นโครงสร้างหลักของเว็บแอปพลิเคชันที่จำเป็นและเป็นมาตรฐานสากล |
| **CSS3** | ใช้สำหรับจัดรูปแบบและดีไซน์หน้าตาของแอปพลิเคชันให้สวยงามและตอบสนองต่อขนาดหน้าจอต่างๆ |
| **JavaScript (ES6+)** | เป็นภาษาหลักที่ใช้ในการพัฒนา Logic ของฝั่ง Frontend ทั้งหมด ตั้งแต่การจัดการ DOM, การโต้ตอบกับผู้ใช้, การควบคุมสถานะของเกม, ไปจนถึงการสื่อสารกับ Backend ผ่าน API |

**เหตุผลโดยรวม:** การใช้ HTML, CSS, และ JavaScript เป็นทางเลือกพื้นฐานที่แข็งแกร่งสำหรับการสร้างเว็บแอปพลิเคชันที่ไม่ซับซ้อน ทำให้เริ่มต้นได้รวดเร็ว, มี Community ที่ใหญ่, และทำงานได้บนทุกเบราว์เซอร์โดยไม่ต้องพึ่งพา Framework ที่มีขนาดใหญ่เกินความจำเป็นของโครงการ

---

### Backend & Infrastructure
| เทคโนโลยี | เหตุผลในการเลือก |
| :--- | :--- |
| **Firebase Hosting** | - **ง่ายต่อการ Deploy:** สามารถ Deploy เว็บแอปพลิเคชัน (ไฟล์ HTML, CSS, JS) ได้ด้วยคำสั่งเดียว<br>- **CDN ในตัว:** ให้บริการไฟล์แก่ผู้ใช้ทั่วโลกได้อย่างรวดเร็ว<br>- **SSL ฟรี:** ตั้งค่า HTTPS ได้โดยอัตโนมัติ เพิ่มความปลอดภัย |
| **Firebase Authentication** | - **รองรับหลาย Providers:** ตอบโจทย์ความต้องการของระบบที่ต้องการให้ล็อกอินได้ทั้งแบบ Email/Password และ Google Account<br>- **ปลอดภัยและจัดการง่าย:** Firebase จัดการเรื่องความปลอดภัยของ Token และ Session ให้ทั้งหมด ลดภาระของนักพัฒนา |
| **Firestore Database** | - **NoSQL & Real-time:** เป็นฐานข้อมูลแบบ NoSQL ที่มีความยืดหยุ่นสูง เหมาะกับการเก็บข้อมูลที่ไม่ตายตัว เช่น ข้อมูลของผู้ใช้, ด่านที่สร้างเอง, และประวัติการเล่น<br>- **Scalability:** สามารถขยายตัวเพื่อรองรับผู้ใช้จำนวนมากได้โดยอัตโนมัติ<br>- **ง่ายต่อการเชื่อมต่อ:** มี SDK สำหรับ JavaScript ที่ช่วยให้การอ่านเขียนข้อมูลจาก Frontend ทำได้สะดวกและปลอดภัยผ่าน Security Rules |

**เหตุผลโดยรวม:** การเลือกใช้ **Firebase** เป็น BaaS ช่วยลดความซับซ้อนในการพัฒนาและดูแลรักษาฝั่ง Server ลงได้อย่างมาก เนื่องจากนักพัฒนาไม่จำเป็นต้องจัดการกับ Server, Database, หรือ Infrastructure เอง ทำให้สามารถมุ่งเน้นไปที่การพัฒนาฟีเจอร์ของแอปพลิเคชันได้อย่างเต็มที่ อีกทั้งยังมี Free Tier ที่เพียงพอต่อการเริ่มต้นและทดสอบโครงการขนาดเล็กอย่าง "Wording" ได้เป็นอย่างดี
